浏览器知识点

发表于 2019-03-13 | 分类于 web前端

事件机制

  1. 捕获阶段 - window往事件触发处传播，遇到注册的捕获事件会触发
  2. 目标阶段 - 传播到触发的事件处，触发注册的事件
  3. 冒泡阶段 - 从事件触发出往window传播，遇到注册的冒泡事件会触发
    如：

捕获阶段开始 -> window -> document -> documentElement -> body -> A -> 捕获阶段结束
目标阶段开始 -> B -> B -> 目标阶段结束
冒泡阶段开始 -> A -> body -> documentElement -> document -> window -> 冒泡阶段结束

注意：通常事件触发会按照这个顺序进行，但当目标节点同时注册捕获和冒泡事件时，事件触发顺序取决于事件注册的顺序。

事件注册：事件注册：target.addEventListener ,该函数的第三个参数可以是布尔值，也可以是对象。若为布尔值，true代表捕获阶段执行，false(默认)代表冒泡阶段执行。

  • 事件代理：
    依赖浏览器的事件冒泡机制，通过将事件添加到它们的父节点而将事件委托给父节点来触发处理函数。

优点：节约内存、无需给子节点注销事件

浏览器环境下的 Event Loop

Javascript是一门单线程的语言，因为它主要用作与用户交互以及DOM操作。如果是多线程，当两个线程同时操作一个DOM，就会出现问题。

部分概念：

  1. 所有同步任务都在主线程上执行，形成执行栈；
  2. 主线程之外，存在一个任务队列。每当异步任务有了结果，会被加入到此队列中；
  3. 待执行栈为空时，Event Loop会从任务队列中取出任务进行执行；
  4. 异步任务分为（微任务micro task和宏任务macro task）；
  5. 主线程会不断重复上面的步骤，直到执行完所有任务。

宏任务: script(整体代码) setTimeout setInterval I/O UI rendering setImmediate(Node.js 环境)
微任务：Promise MutaionObserver(HTML5 新特性) process.nextTick(Node.js 环境)

浏览器渲染

浏览器显示一个页面，经历过程：

  1. DNS查询
  2. TCP连接
  3. 发送HTTP请求
  4. 服务器响应
  5. 浏览器解析渲染页面
  6. 连接结束

其中第五点，浏览器解析渲染页面，主要分为以下步骤：

  1. 处理HTML标记，构建DOM树
  2. 处理CSS标记，构建CSSOM树
  3. 将DOM和CSSOM合并成渲染树
  4. 根据渲染树来布局，以计算每个节点的几何信息
  5. 将每个节点绘制到屏幕

在构建 CSSOM 树时，会阻塞渲染，直至 CSSOM 树构建完成。
当 HTML 解析到 script 标签时，会暂停构建 DOM，完成后才会从暂停的地方重新开始。
当存在 defer 和 async，script 的加载顺序会有所不同。
备注：蓝色线代表网络读取，红色线代表执行时间，这俩都是针对脚本的；绿色线代表 HTML 解析。

页面生命周期：

  • DOMContentLoaded 事件在DOM树构建完毕后被触发。
    • async和defer的脚本可能还没有执行。
    • 图片及其他资源文件可能还在下载中。
  • load 事件在页面所有资源被加载完毕后触发。
  • beforeunload 在用户即将离开页面时触发，返回一个字符串，浏览器会展示并询问这个字符串以确定是否离开。
  • unload 在用户已经离开时触发，我们在这个阶段仅可以做一些没有延迟的操作，由于种种限制，很少被使用。
  • document.readyState 表征页面的加载状态，可以在readystatechange中追踪页面的变化状态：
    • loading — 页面正在加载中。
    • interactive – 页面解析完毕，时间上和 DOMContentLoaded同时发生，不过顺序在它之前。
    • complete – 页面上的资源都已加载完毕，时间上和window.onload同时发生，不过顺序在他之前。

回流与重绘（Reflow & Repaint）

  • 对于元素视觉上的改变(如改变outline，background-color，visibility等)会触发repaint；
  • 对于元素布局上的改变(增删节点，触发resize事件，修改style属性等)会触发reflow；

回流产生：

  1. 添加或者删除可见的DOM元素；
  2. 元素位置改变；
  3. 元素尺寸改变——边距、填充、边框、宽度和高度
  4. 内容改变——比如文本改变或者图片大小改变而引起的计算值宽度和高度改变；
  5. 页面渲染初始化；
  6. 浏览器窗口尺寸改变——resize事件发生时；

回流一定会触发重绘，而重绘不一定会回流。

最小化重绘和重排：

  1. 不要一个一个改变元素的样式属性，直接改变className，如果动态改变样式，则使用cssText。
  2. 让要操作的元素进行“离线处理”，处理完后一起更新。这里所谓的”离线处理”即让元素不存在于render tree中。如：DocumentFragment, display:none只引发两次回流和重绘，cloneNode(true or false) 和 replaceChild
  3. 不要经常访问会引起浏览器flush队列的属性，如果你确实要访问，利用缓存。
  4. 让元素脱离动画流，减少回流的Render Tree的规模。

Session、Cookie、sessionStorage、localStorage

  • Session 存储在服务器端

Session是在服务端保存的一个数据结构，运行依赖Session ID，用来跟踪用户的状态。Session ID一般存在于Cookie中。浏览器关闭后这次的Session就消失了（变了）。

  • Cookie sessionStorage localStorage 存储在客户端

Cookie 大小限制4KB左右，一般由服务器生成，可设置失效时间。如果在浏览器端生成Cookie，默认是关闭浏览器后失效。每次都会携带在HTTP头中，如果使用cookie保存过多数据会带来性能问题，可以通过Cookie隔离，把部分静态资源放在非主域名下，这样在请求这些资源时就不会带上主域名的cookie，降低传输成本。

参数设置：
Expires=<date> 过期时间
Max-Age=<non-zero-digit> 生成后失效秒数
Domain=<domain-value> 可送达的主机域名
Path=<path-value> 请求的资源 URL
Secure 必须使用SSL或HTTPS协议
HttpOnly 不能通过 JS 访问 Cookie，减少 XSS 攻击

sessionStorage、localStorage：
大小限制5M左右，都有同源策略。
不同点：
sessionStorage：仅在当前会话窗口有效。不同窗口不能共享数据，当
localStorage：创建后一直存在。同一个浏览器中同源文档可以共享。同一个窗口有两个同源的iframe元素的话，这两个iframe的sessionStorage 是可以互通的。

浏览器缓存

浏览器缓存分析：

强制缓存

当浏览器向服务器发起请求时，服务器会将缓存规则放入HTTP响应报文的HTTP头中和请求结果一起返回给浏览器，控制强制缓存的字段分别是Expires和Cache-Control，其中Cache-Control优先级比Expires高。

  • Expires
    Expires是HTTP/1.0控制网页缓存的字段，其值为服务器返回该请求结果缓存的到期时间，即再次发起该请求时，如果客户端的时间小于Expires的值时，直接使用缓存结果。
    由于可能存在客户端与服务器因为某些原因（如：时差）发生误差，基本不再使用。

  • Cache-Control
    在HTTP/1.1中，主要有以下取值：

    • public : 所有内容都将被缓存（客户端和代理服务器都可缓存）
    • private : 所有内容只有客户端可以缓存，Cache-Control的默认取值
    • no-cache : 客户端缓存内容，但是是否使用缓存则需要经过协商缓存来验证决定
    • no-store : 所有内容都不会被缓存，即不使用强制缓存，也不使用协商缓存
    • max-age=xxx (xxx is numeric) : 缓存内容将在xxx秒后失效

  • 缓存位置

内存缓存(from memory cache)和硬盘缓存(from disk cache)

内存缓存读取快，但是当进程关闭，该进程的内存会被清空。

协商缓存

协商缓存就是强制缓存失效后，浏览器携带缓存标识向服务器发起请求，由服务器根据缓存标识决定是否使用缓存的过程。

  • Last-Modified / If-Modified-Since

    Last-Modified 是服务器响应请求时，返回该资源文件在服务器最后被修改的时间。
    If-Modified-Since 则是客户端再次发起该请求时，携带上次请求返回的Last-Modified值。
    服务器对比该字段和资源在服务器最后修改的时间，返回200和新资源或304。

  • Etag / If-None-Match

Etag是服务器响应请求时，返回当前资源文件的一个唯一标识(由服务器生成)。
If-None-Match是客户端再次发起该请求时，携带上次请求返回的唯一标识Etag值。
服务器对比该字段和该资源在服务器的Etag值，返回200和新资源或304。

Etag / If-None-Match的优先级比Last-Modified / If-Modified-Since高。

缓存总结

强制缓存优先于协商缓存进行，若强制缓存(Expires和Cache-Control)生效则直接使用缓存，若不生效则进行协商缓存(Last-Modified / If-Modified-Since和Etag / If-None-Match)，协商缓存由服务器决定是否使用缓存，若协商缓存失效，那么代表该请求的缓存失效，重新获取请求结果，再存入浏览器缓存中；生效则返回304，继续使用缓存。

跨域问题

只要协议、域名、端口有任何一个不同，就是跨域。

浏览器有一个同源策略，用来保护用户的安全。

  • DOM同源策略：禁止对不同源页面DOM进行操作。这里主要场景是iframe跨域的情况，不同域名的iframe是限制互相访问的。
  • XmlHttpRequest同源策略：禁止使用XHR对象向不同源的服务器地址发起HTTP请求。

解决办法：

  • JSONP
  • CORS
  • 代理
  • document.domain
  • window.name
  • location.hash
  • postMessage
